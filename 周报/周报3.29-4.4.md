# 周报（3.29–4.4）

## 一、本周目标

1. 修改或重写 `myl7` 项目的 benchmark，实现对 bin、bout 及测试规模等参数的可控配置。
2. `Orca` 增加批大小好像不能等价于提高 GPU 并行能力，批大小从一百万到一千万这一块，基本上是线性增长的，似乎没有起到加速作用
3. `Orca` 中每次评估的密钥都是重新生成的，如果允许所有评估都复用同一个密钥，能不能提高评估阶段的性能，减少CPU、GPU间数据传输的开销
4. 在相同 GPU 和 CPU 硬件条件下，比较 `myl7` 的 GPU / CPU 优化倍数以及 `Orca` / `libfss`的优化倍数，如果 `myl7` 的优化倍数更多，进一步分析一下原因

## 二、本周完成情况

1. 对 `Orca` 的批大小增长现象做了整理。从 `10^6` 到 `10^7` 基本线性增长是正常的，不是 GPU 没有并行，而是这个量级下单个元素的计算成本已经比较稳定，继续增大 `n` 主要是在增加总实例数和 `CPU/GPU` 间的数据搬运量，所以总时间会接近线性增长。

2. 将 `Orca` 复用密钥版、`myl7` 和 `libfss` 的 benchmark 整理到了 `/FSS/重构对比`，统一了构建方式和输出格式；其中需要 GPU 参数的部分会尽量自动读取本机环境。当前测试统一参数为：`DPF bin=64`、`DCF bin=64,bout=1`。

| 环境 | 信息 |
| --- | --- |
| GPU | `NVIDIA L20`（`49140 MiB`） |
| CPU | `Intel(R) Xeon(R) Gold 6462C` |
| CUDA | `12.8` |
| CMake | `3.22.1` |

在 `N=100000` 条件下，`myl7` 和重构后的 `Orca`（复用密钥）对比结果如下：

| 函数 | 实现 | `keygen(us)` | `setup_p0/p1(us)` | `eval_p0/p1(us)` | `transfer_p0/p1(us)` | `total(us)` |
| --- | --- | ---: | ---: | ---: | ---: | ---: |
| `DPF` | `Orca` 复用密钥 | `46710` | `3827 / 3818` | `469 / 426` | `-` | `55357` |
| `DPF` | `myl7` | `5760` | `-` | `932 / 887` | `148 / 142` | `11127` |
| `DCF` | `Orca` 复用密钥 | `56198` | `3822 / 3823` | `727 / 678` | `-` | `65320` |
| `DCF` | `myl7` | `5447` | `-` | `932 / 880` | `157 / 143` | `10888` |

其中 `Orca` 的 `setup_p0/p1` 表示两方密钥首次上传到 GPU 的时间。可以看出，复用密钥后 `Orca` 复用密钥确实减少了 `CPU/GPU` 间的数据传输；但在当前这组参数下，`myl7` 的总时间仍然更短。
