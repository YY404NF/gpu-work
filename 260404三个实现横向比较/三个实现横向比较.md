# 在相同 GPU 和 CPU 硬件条件下，比较 `myl7` 的 GPU / CPU 优化倍数以及 `Orca` / `libfss`的优化倍数，如果 `myl7` 的优化倍数更多，进一步分析一下原因

## 工作记录

### 1. 本次测试使用的硬件

### 2. 对比口径

这次没有直接拿三边现成 benchmark 生硬相减，而是先统一了“测什么”：

### 7. 结论

这次在 `bin=20, n=131072` 这组参数下，结果是：

- `Orca/libfss` 的优化倍数比 `myl7 GPU/CPU` 更大
- 所以“如果 `myl7` 的优化倍数更多，再进一步分析原因”这个条件在这次实测里没有触发

### 8. 但还需要补一句：倍数更大，不代表绝对性能更强

这次最容易误读的地方就在这里。

虽然 `Orca/libfss` 的倍数更大，但 `myl7` 的绝对时间其实更快：

| primitive | `myl7_cpu` vs `libfss_cpu` | `myl7_gpu` vs `Orca total` |
| --- | ---: | ---: |
| `DPF` | `libfss / myl7 = 9.77x` | `Orca / myl7 = 2.28x` |
| `DCF` | `libfss / myl7 = 7.15x` | `Orca / myl7 = 2.40x` |

也就是说：

- `myl7` 的 CPU 本身就已经比 `libfss` 快很多
- `myl7` 的 GPU 绝对时间也比 `Orca` 更短
- 只是因为 `myl7` 的 CPU baseline 已经很强，所以它的“GPU/CPU 倍数”不会像 `Orca/libfss` 那样夸张

### 9. 为什么会出现这个现象

我觉得这次结果最合理的解释有 4 点：

1. `libfss` 的 CPU baseline 更老，也更慢  
   `Orca/libfss` 的分母是 `libfss`，而 `libfss` 在这组测试里明显慢于 `myl7_cpu`，所以 `Orca/libfss` 的倍数天然更容易做大。

2. `myl7` 是“CPU 和 GPU 两边都做得不错”，不是“CPU 很慢、GPU 很快”  
   从结果看，`myl7_cpu` 并不弱，所以它的 speedup ratio 被压低了；这不是坏事，反而说明它 CPU 端实现也比较强。

3. `myl7` 的 GPU 绝对性能确实很强，尤其是 eval  
   如果只看两边 GPU 相关阶段：
   - `DPF` 的 `myl7` GPU 总时间比 `Orca` 快 `2.28x`
   - `DCF` 的 `myl7` GPU 总时间比 `Orca` 快 `2.40x`
   - 只看 `eval_p0 + eval_p1`，`myl7` 比 `Orca` 快大约 `11x`

4. 这次比较的是“项目级 end-to-end speedup”，不是“同一段 kernel 的 isolated speedup”  
   `myl7` 这边我已经把 `H2D/D2H` 算进 total 了，但它仍然没有 `Orca` 那种 `KeyBlob` 序列化/反序列化开销。  
   即便如此，`Orca/libfss` 的倍数还是更大，说明主要差异不在 GPU 端，而在 CPU baseline：`libfss` 实在比 `myl7_cpu` 慢很多。

### 10. 一个补充观察

如果只看“单个项目内部的关键阶段 speedup”：

| primitive | `myl7 keygen speedup` | `myl7 eval speedup` | `Orca/libfss keygen speedup` | `Orca/libfss eval speedup` |
| --- | ---: | ---: | ---: | ---: |
| `DPF` | `33.09x` | `530.97x` | `138.88x` | `128.87x` |
| `DCF` | `67.52x` | `1098.09x` | `190.61x` | `161.64x` |

这里能看到：

- `myl7` 在 `eval` 这一段的项目内 speedup 非常夸张
- 但总倍数还是被 `CPU baseline 已经很强` 以及 `GPU 侧仍有搬运/分配成本` 拉回来了

所以，单看 speedup ratio，会得出 `Orca/libfss` 更大；但如果看绝对性能，`myl7` 反而是这次测试里更快的一套实现。
