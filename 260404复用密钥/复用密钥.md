# `Orca` 中每次评估的密钥都是重新生成的，如果允许所有评估都复用同一个密钥，能不能提高评估阶段的性能，减少CPU、GPU间数据传输的开销

## 2026-04-04 更正后的研究记录

上一次我误看到了 `GPU-MPC/Orca` 路径，那部分与本题不对应。下面这版以 `/FSS` 里的 `dpf/dcf benchmark` 为准，前面的分析以本节为准。

### 1. 先核实当前 `/FSS` benchmark 的实际行为

#### 1.1 `dpf_benchmark`

- 当前实现是一趟 `keygen`，然后各做一次 `SERVER0` / `SERVER1` 的 `eval`。
- 代码位置：
  - `FSS/dpf_benchmark.cu:79-82`
  - `FSS/dpf_benchmark.cu:87-111`

也就是说：

- 单次 benchmark 运行里，并不存在“每次 eval 之前都重新生成 key”的循环。
- 现在的程序结构本身就是：
  - 先 `generateDpfKeys(...)`
  - 再 `gpuDpf(...)` 求值一次

#### 1.2 `dcf_benchmark`

- `DCF` benchmark 的结构完全一样：一趟 `keygen`，然后 `P0/P1` 各做一次求值。
- 代码位置：
  - `FSS/dcf_benchmark.cu:87-90`
  - `FSS/dcf_benchmark.cu:95-119`

阶段性结论：

- 对“当前这两个 benchmark 的一次运行”来说，`key` 本来就不是在 `eval` 前反复生成的。
- 如果想研究“复用同一把 key”带来的收益，需要人为把 benchmark 改成“`keygen` 一次，`eval` 多次”的结构，当前版本还没有这个场景。

### 2. 当前 benchmark 里，`eval_p0/eval_p1` 到底统计了什么

- `eval_p0/eval_p1` 的计时只包住了 `gpuDpf(...)` / `gpuDcf(...)` 这次调用本体。
- `x` 的上传发生在计时前。
- 输出从 GPU 拷回 CPU 发生在计时后。
- 代码位置：
  - `FSS/dpf_benchmark.cu:84-98`
  - `FSS/dpf_benchmark.cu:103-111`
  - `FSS/dcf_benchmark.cu:92-106`
  - `FSS/dcf_benchmark.cu:111-119`

- `transfer_p0/transfer_p1` 来自 `Stats::transfer_time`。
- `moveToGPU / moveToCPU / moveIntoGPUMem / moveIntoCPUMem` 才会往这个字段里累计时间。
- 代码位置：
  - `FSS/gpu/gpu_mem.cu:87-130`

这意味着：

- `transfer_p0/transfer_p1` 不是完整的端到端 CPU/GPU 传输时间。
- 它主要反映的是 `gpuDpf/gpuDcf` 调用内部的传输。
- 当前 benchmark 里，`x` 的上传和输出拷回主机都没有记进 `transfer_p0/transfer_p1`。

### 3. `KeyBlob` 复用到哪一层，当前代码是怎么做的

- `generateDpfKeys / generateDcfKeys` 产出的是 `KeyBlob`，也就是一段序列化后的 host 侧字节流。
- `KeyBlob` 本身分配在 pinned host memory 上。
- 代码位置：
  - `FSS/fss/dpf_api.h:17-39`
  - `FSS/fss/dcf_api.h:17-39`
  - `FSS/runtime/standalone_runtime.h:158-205`

- `readGPUDPFKey / readGPUDCFKey` 并不会把 key 变成 GPU 常驻对象。
- 它们只是把这段 host blob 切成多个“只读视图”。
- 代码位置：
  - `FSS/fss/gpu_dpf.h:28-83`
  - `FSS/fss/gpu_dcf.h:26-80`

所以当前层次上所谓“复用 key”，实际最多只是：

- 复用同一份 host 侧 `KeyBlob`
- 以及复用同一份 host 侧 `parsedKey` 视图

但这还不是 GPU 侧复用。

### 4. 真正的 `eval` 里，key 依然会每次从 CPU 重新搬到 GPU

#### 4.1 DPF tree 路径

- 在 `gpuDpfTreeEval(...)` 里，`scw / l0 / l1 / tR` 会在每次求值时重新 `moveToGPU(...)`。
- kernel 结束后又会 `gpuFree(...)`。
- 代码位置：
  - `FSS/fss/gpu_dpf.cu:212-230`

#### 4.2 DCF tree 路径

- 在 `gpuDcfTreeEval(...)` 里，`scw / vcw / l` 同样会在每次求值时重新上传。
- 代码位置：
  - `FSS/fss/gpu_dcf.cu:125-140`

#### 4.3 小 `bin` 的 SSTable 路径也一样

- `DPF` 小 `bin` 走 `gpuLookupSSTable(...)`，每次 eval 都会把 `ss` 这张表重新 `moveToGPU(...)`。
- 代码位置：
  - `FSS/fss/gpu_sstab.h:64-74`

- `DCF` 小 `bin` 的 `gpuLookupSSTable(...)` 也是同样行为。
- 代码位置：
  - `FSS/fss/gpu_dcf_sstab.h:37-47`

因此：

- 无论是 tree key 还是 SSTable key，当前 `/FSS` eval 路径都不是“GPU 侧复用 key”。
- 现在的实现是“host 侧存 key，eval 时临时传到 GPU，用完释放”。

### 5. 这对“复用密钥能不能提速”意味着什么

需要分成两个层面看。

#### 5.1 只复用同一份 host `KeyBlob`

如果把 benchmark 改成：

- `keygen` 一次
- 对同一份 key 连续做很多次 `eval`

那么这种“复用”能带来的收益主要是：

- 不再重复做 `keygen`

但它不能自动消掉的开销是：

- 每次 `eval` 内部仍然会把 key 材料从 CPU 重新搬到 GPU

所以：

- 对当前 `transfer_p0/transfer_p1` 这个统计项来说，单纯复用 host key，理论上不会把它压到很低。
- 对 `eval_p0/eval_p1` 来说，也只能间接减少极少的 host 侧解析开销，不能消掉主要的 H2D key 上传。

#### 5.2 复用 GPU 侧 key

如果进一步做成：

- key 在第一次 eval 前上传到 GPU
- 后续 eval 只复用 GPU resident 的 `scw/l0/l1/tR` 或 `scw/vcw/l` / `ss`

那么这时才会真正减少：

- `gpuDpf/gpuDcf` 内部的 H2D 传输
- 也就是当前 benchmark 里 `transfer_p0/transfer_p1` 主要对应的那部分时间

所以真正有价值的方向是：

- 不是 `host key reuse`
- 而是 `device-side key cache`

### 6. 还有一个容易误读的点：当前 benchmark 的 `x` 上传不在 `transfer_p0/p1` 里

- `x` 是通过 `detail::copyVectorToGpu(x)` 上传的。
- 这里传给 `moveToGPU(...)` 的 `Stats*` 是 `nullptr`。
- 代码位置：
  - `FSS/runtime/standalone_runtime.h:108-116`

所以如果后面做复用实验，要分清楚：

- 当前统计里的 `transfer_p0/p1`，主要是 key 的 H2D
- 不是输入 `x` 的 H2D

这也解释了为什么本题里“复用 key”最直接影响的是 key upload，而不是 query upload。

### 7. 当前针对 `/FSS` benchmark 的准确结论

#### 7.1 对“现有 benchmark 是否每次 eval 都重生 key”

- 不是。
- 当前 `dpf_benchmark/dcf_benchmark` 在单次运行里，本来就是 `keygen` 一次、`eval` 一次。

#### 7.2 对“如果允许所有 eval 复用同一把 key，能不能提高 eval 性能”

- 如果只是复用同一份 host `KeyBlob`：
  - 可以摊薄未来多轮 benchmark 的 `keygen` 成本；
  - 但不能消掉当前 `eval` 内部每次都发生的 key H2D 传输。

- 如果要真正减少 `eval` 阶段的 CPU/GPU 传输：
  - 必须把 key 做成 GPU 常驻缓存；
  - 否则 `scw/l0/l1/tR`、`scw/vcw/l` 或 `ss` 仍然会在每次 eval 时重新上传。

#### 7.3 对“当前打印出来的 `eval_p0/eval_p1/transfer_p0/transfer_p1` 会不会立刻变好”

- 在不改 benchmark 结构的前提下，不会。
- 因为当前 benchmark 只做一次 eval，根本没有“第二次起复用作用”的机会。

### 8. 下一步如果继续做，最合理的实验设计

建议按三组实验来做：

1. `baseline`
   - 保持当前 benchmark 结构

2. `host-key-reuse`
   - `keygen` 一次
   - 对同一份 host `KeyBlob` 连续做 `K` 次 `eval`
   - 观察平均 `eval_p0/eval_p1` 与 `transfer_p0/transfer_p1`

3. `device-key-reuse`
   - `keygen` 一次
   - key 只上传一次到 GPU
   - 连续做 `K` 次 `eval`
   - 与第 2 组比较

如果后面允许改 `/FSS` 源码，优先切入点就是：

- `FSS/fss/gpu_dpf.cu`
- `FSS/fss/gpu_dcf.cu`
- `FSS/fss/gpu_sstab.h`
- `FSS/fss/gpu_dcf_sstab.h`

### 9. 当前一句话结论

- 对 `/FSS` 现有 `dpf/dcf benchmark` 来说，单次运行里 key 本来就只生成一次；
- 真正阻碍 eval 复用收益的，不是 keygen，而是每次 eval 都把 key 从 CPU 重新搬到 GPU；
- 所以想减少评估阶段的 CPU/GPU 传输，下一步该研究的是 `GPU 常驻 key cache`，而不是只复用同一份 host key。
