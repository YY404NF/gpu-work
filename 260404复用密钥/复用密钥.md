# `Orca` 中每次评估的密钥都是重新生成的，如果允许所有评估都复用同一个密钥，能不能提高评估阶段的性能，减少CPU、GPU间数据传输的开销

## 2026-04-04 更正后的研究记录

当前只关注 `/FSS` 的 `dpf_benchmark.cu` / `dcf_benchmark.cu`。

### 1. 当前基线结论

- 现有 benchmark 单次运行里本来就是 `keygen` 一次、`eval` 一次，不存在“每次 eval 前重生 key”的循环。
- `transfer_p0/transfer_p1` 只统计 `gpuDpf/gpuDcf` 内部的传输，不包含 `x` 的上传和输出拷回主机。
- `KeyBlob` 只是 host 侧序列化 key；`readGPUDPFKey/readGPUDCFKey` 只是切 host 视图，不会把 key 变成 GPU 常驻对象。
- 真正的瓶颈在于 eval 时仍会把 key 重新 `moveToGPU(...)`：
  - `DPF tree`: `FSS/fss/gpu_dpf.cu:212-230`
  - `DCF tree`: `FSS/fss/gpu_dcf.cu:125-140`
  - `SSTable`: `FSS/fss/gpu_sstab.h:64-74`，`FSS/fss/gpu_dcf_sstab.h:37-47`

一句话结论：

- 只复用 host 侧 key，能省的是未来多轮实验里的 `keygen`；
- 真正想把 `eval` 阶段的 CPU/GPU 传输降下来，必须做 `device-side key cache`。

### 2. 当前目录里的新 benchmark

已在当前目录新增：

- `dpf_benchmark_reuse.cu`
- `dcf_benchmark_reuse.cu`

实现方式：

- `keygen` 一次
- 两方 key 各自只上传一次到 GPU
- 同一份 GPU 常驻 key 连续做 `N` 次 eval
- 每次 eval 后只释放输出 buffer，不重复上传 key
- 额外做了一次正确性校验：
  - cached-key 输出与现有 `/FSS` 路径输出逐位一致
  - 再检查两方 share 合并后的语义

说明：

- `DCF` 小 `bin` 路径当前实现更接近 `<`，而不是文档里描述的 `<=`；本地校验已按现有实现对齐。
